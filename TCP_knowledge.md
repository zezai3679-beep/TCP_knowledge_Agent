# TCP协议的三次握手和四次挥手

## 什么是TCP？

TCP（Transmission Control Protocol，传输控制协议）是互联网核心协议之一，位于OSI七层模型中的传输层。它是一种面向连接的、可靠的字节流传输服务。

想象一下，你在网上购物下单。商品从卖家到你手里，需要快递公司可靠地把货物送到。TCP就像那个保证货物安全送达的快递公司——它确保数据完整、有序、不丢失地从一台设备传输到另一台设备。

## TCP三次握手：建立连接的过程

在TCP传输数据之前，必须先建立连接。这个建立连接的过程需要三个步骤，所以叫做"三次握手"。

### 为什么要握手？

试想一下，如果你给朋友寄一封信，你不知道朋友的地址是否有效，也不知道他是否在线。TCP握手就是为了确认双方都能正常通信，就像打电话前要先确认对方号码能打通、对方在线一样。

### 握手的三个步骤

**第一步：客户端发送SYN**

客户端向服务器发送一个SYN（Synchronize）报文，请求建立连接。这个报文里有一个重要的数字——初始序列号（ISN，Initial Sequence Number）。序列号就像是给每个数据包编的序号，用来确保数据按顺序到达。

```
客户端 服务器
    | |
    |----------------- SYN, seq=x ------------------>|
    |            我想建立连接，这是我的初始序号 |
```

**第二步：服务器回复SYN+ACK**

服务器收到SYN后，如果同意建立连接，会回复一个SYN+ACK报文。SYN表示服务器也把自己的初始序列号发给客户端，ACK表示确认收到了客户端的SYN。

```
客户端                                              服务器
    |                                                 |
    |<-------- SYN+ACK, seq=y, ack=x+1 --------------|
    |            收到，我同意， 这是我的初始序号        |
```

**第三步：客户端发送ACK**

客户端收到SYN+ACK后，再发送一个ACK确认报文给服务器。这个报文中ack=x+1表示"我收到了你的SYN+ACK"。

```
客户端                                              服务器
    |                                                 |
    |----------------- ACK, ack=y+1 ---------------->|
    |            收到确认，连接建立完成 |
```

### 三次握手的本质

你可能会问：为什么非要三次？

一次不够吗？——一次的话，客户端发了SYN就以为连接建立了，但服务器可能根本没收到。
两次呢？—— 服务器回复SYN+ACK后，客户端收到确认连接就建立了。但服务器不知道自己的SYN+ACK客户端有没有收到。
三次刚刚好—— 第三次ACK发送后，客户端和服务器都确认双方收发能力正常。

## TCP四次挥手：关闭连接的过程

当数据传输完成后，需要优雅地关闭连接。四次挥手确保双方都把数据传完了才断开。

### 为什么要四次？

关闭连接不像开门关门那么简单。数据传输是双向的，客户端说"我发完了"，服务器可能还在给客户端发数据。所以关闭连接需要双向分别确认。

### 挥手的四个步骤

**第一步：主动关闭方发送FIN**

假设客户端想关闭连接。客户端会发送一个FIN（Finish）报文，表示"我这边数据发完了"。

```
客户端                                              服务器
    |                                                 |
    |----------------- FIN, seq=u ------------------>|
    |            我发完了，等你确认 |
```

**第二步：服务器回复ACK**

服务器收到FIN后，回复一个ACK确认。这只是说"收到了"，但服务器可能还在发数据。

```
客户端                                              服务器
    |                                                 |
    |<----------- ACK, ack=u+1 --------------------|
    |            收到，但可能还在给你发数据            |
```

**第三步：服务器发送FIN**

服务器把数据发完后，也发送一个FIN给客户端，表示"我这边也发完了"。

```
客户端                                              服务器
    |                                                 |
    |<----------- FIN, seq=w -----------------------|
    |            我也发完了，准备关闭                  |
```

**第四步：客户端回复ACK**

客户端收到服务器的FIN后，回复最后一个ACK确认。

```
客户端                                              服务器
    |                                                 |
    |----------------- ACK, ack=w+1 ---------------->|
    |            收到，连接正式关闭                    |
```

### 挥手后的等待

客户端发送最后一个ACK后，不会立即关闭，而是进入TIME_WAIT状态，等待2MSL（Maximum Segment Lifetime，最大报文生存时间）才真正关闭。这是为了防止最后一个ACK丢失，服务器重发FIN时客户端还能响应。

## 状态转换图

### 客户端状态变化

- **CLOSED**：初始状态
- **SYN_SENT**：发了SYN，等待回复
- **ESTABLISHED**：连接建立成功，可以传输数据
- **FIN_WAIT_1**：发了FIN，等待ACK
- **FIN_WAIT_2**：收到了ACK，等待服务器的FIN
- **TIME_WAIT**：收到了FIN的ACK，等待2MSL后关闭

### 服务器状态变化

- **LISTEN**：服务器在监听端口
- **SYN_RECEIVED**：收到了SYN，回复了SYN+ACK
- **ESTABLISHED**：连接建立成功
- **CLOSE_WAIT**：收到了FIN，等待应用程序处理
- **LAST_ACK**：应用程序处理完，发送了FIN，等待最后一个ACK

## 实际抓包示例

如果你想亲眼看看三次握手和四次挥手，可以用Wireshark这款网络抓包工具。抓包时过滤条件写`tcp.flags.syn==1`看握手，`tcp.flags.fin==1`看挥手。

抓包看到的三次握手大概长这样：
- 第一个包：SYN，seq=0
- 第二个包：SYN+ACK，seq=0，ack=1
- 第三个包：ACK，seq=1，ack=1

## 常见面试问题

### Q1：为什么是三次握手而不是两次？

两次握手无法防止历史连接初始化混乱，也无法同步双方初始序列号。比如客户端发了SYN超时重传了，服务器收到两个SYN，无法判断哪个是历史连接。

### Q2：为什么是四次挥手而不是三次？

因为TCP是全双工协议，每个方向都需要单独关闭。服务器收到FIN后可能还有数据要发，所以先回ACK，等数据发完了再发自己的FIN。

### Q3：SYN洪水攻击是什么？

攻击者大量发送SYN包但不完成握手，服务器维护大量半开连接，耗尽资源。防御方法有SYN Cookie、缩短SYN Timeout等。

### Q4：TIME_WAIT状态有什么作用？

确保旧连接的所有数据包都在网络中消失，防止影响新连接。2MSL通常为60秒。

## 总结

TCP的三次握手和四次挥手是网络通信中的经典机制。三次握手确保双方确认彼此的发送和接收能力，建立可靠连接；四次挥手确保双方数据都传完，优雅关闭连接。理解这两个过程，对于学习网络协议、排查网络问题、应对面试都非常重要。
